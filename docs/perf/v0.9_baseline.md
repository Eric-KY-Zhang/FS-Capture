# v0.9 Baseline Performance

**Date**: 2026-05-24
**Build under test**: v0.9.0 codebase after SG plugin/UI Checkpoint A, before performance optimizations
**Command**: `FS_CAPTURE_RUN_BENCHMARK=1 FS_CAPTURE_BENCHMARK_RUNS=1 pytest -m benchmark --benchmark-only -s`
**Environment**: Windows 11 / Python 3.14.3 / public WAN sources / default `max_workers = 4`

> Note: `RUNS=1` was used for this baseline because a full `RUNS=3` pass would multiply already slow network downloads. The benchmark harness supports `FS_CAPTURE_BENCHMARK_RUNS=3` for Reviewer reruns.

## Scenario 3: Batch Download

The literal 24-task run requires JP; the current workstation has no EDINET API key, so JP was excluded. A first unfiltered run timed out at 30 minutes because JP repeatedly hit EDINET `invalid subscription key`.

| Scope | Tasks | Workers | Elapsed (s) | Reports | Bytes | Zero-report Tasks |
|---|---:|---:|---:|---:|---:|---|
| A/HK/US/KR/TW/UK/SG, 3 tickers each | 21 | 4 | 211.76 | 20 | 137,802,891 | UK AZN |

Observed baseline issue: concurrent Playwright rendering can fail with `It looks like you are using Playwright Sync API inside the asyncio loop`; this affected `UK AZN` in the batch run. This is the baseline for Sprint 5.4 Playwright context pooling.
